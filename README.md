QA Automation Practice: UI + API + SQL + Mocking
Комплексный фреймворк для автоматизированного тестирования, объединяющий тестирование фронтенда высоконагруженных платформ, изоляцию интерфейсов через эмуляцию бэкенда (API Mocking), интеграционные тесты серверной логики и валидацию данных в реляционной базе данных.

Проект полностью упакован в Docker и оптимизирован для стабильного запуска в CI/CD пайплайнах без необходимости наличия графического окружения (Headless mode).

🚀 Основные фичи фреймворка
UI Automation (Wildberries & Demoblaze): Автоматизация сложных сценариев интернет-магазинов с динамической версткой, асинхронным рендерингом и управлением сессиями.

Реляционный SQL Бэкенд (SQLite3): Управление состоянием данных (Data State) на лету. Генерация уникальных пользователей, запись транзакций через INSERT и параметризованная выборка SELECT прямо внутри тестовых циклов.

API Mocking & Трафик-инкубатор: Перехват сетевых запросов браузера через page.route(). Изоляция фронтенда от внешних серверов и обход антифрод-защиты маркетплейсов путем подмены ответов на кастомные JSON-структуры.

Интеллектуальный Page Object: Универсальные методы авторизации с динамическими флагами ожидания (expect_success), позволяющие обслуживать как Happy Path, так и негативные сценарии безопасности без падения по таймаутам.

CI/CD Ready & Docker: Полная изоляция тестового окружения внутри Docker-контейнера с флагами оптимизации Chromium (--no-sandbox, --disable-dev-shm-usage).

🛠 Технологический стек
Python 3.12+ — базовый язык программирования.

Pytest — основной движок тестов, управление фикстурами и параметризацией.

Playwright (Python) — современный инструмент UI-автоматизации (работает напрямую через CDP-протокол, обеспечивая встроенные авто-ожидания элементов).

SQLite3 — локальная БД для хранения и валидации сущностей.

Allure Report — интерактивная визуализация результатов и шагов (@allure.step).

Docker — контейнеризация и обеспечение воспроизводимости прогонов.

🛒 Сценарии тестирования и автоматизации
1. Интеграция UI и Реляционной БД (Demoblaze)
test_full_user_lifecycle: Динамическое создание уникального пользователя ismail_*** ➡️ Регистрация через UI ➡️ Отправка SQL-запроса INSERT в базу данных ➡️ Извлечение учетных данных из БД через SELECT ➡️ Успешная авторизация и проверка приветствия на экране.

test_login_banned_user: Негативный сценарий. Извлечение пользователя со статусом banned ➡️ Попытка входа с отключенным ожиданием успеха (expect_success=False) ➡️ Проверка того, что форма осталась открытой, а кнопка входа не исчезла.

2. Изоляция и API Mocking (Wildberries)
test_basket_item: Тестирование добавления товара в корзину (Xiaomi / Iphone). Чтобы изолировать тест от капчи и пустой региональной выдачи в Docker, настроен перехват трафика по маске /lk-basket/. Браузер получает фейковый JSON-ответ с заданным количеством товаров (count: 1), что гарантирует стабильность ассертов.

test_search_input: Параметризованный поиск по различным классам эквивалентности (кириллица, латиница, бренды, спецсимволы).

## 📊 Результаты нагрузочного тестирования (Locust)

В ходе профилирования API бэкенда (`https://api.demoblaze.com`) была проведена сессия тестирования под нагрузкой в **100 одновременных виртуальных пользователей** со случайным временем ожидания (think time) между запросами от 1 до 3 секунд.

* **Производительность:** Система продемонстрировала стабильную среднюю пропускную способность **~40 RPS (Requests Per Second)** при распределении задач согласно бизнес-весам (75% — просмотр каталога `/entries`, 25% — фильтрация по категориям `/bycat`).
* **Стабильность:** Уровень отказов составил **0% (Failures: 0)** на объеме свыше 4500+ запросов.
* **Временные метрики (Response Time):** * Медианное время ответа (50% перцентиль) составило стабильные **360 ms**.
  * Выявлена точка деградации производительности на пиках: 99-й перцентиль увеличился до **1300 ms**, а максимальное зафиксированное время ожидания составило **1622 ms**, что указывает на возникновение очередей на стороне сервера при обработке «тяжелых» POST-запросов к БД.
